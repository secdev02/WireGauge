# WireGuard Protocol Fuzzing Harness

A three-phase kernel-level fuzzing harness for the Linux WireGuard driver, built
from a full security audit of `drivers/net/wireguard/`. It sends crafted and
mutated packets through all three lifecycle stages of a WireGuard session and
records every kernel message, packet, and anomaly for peer review.

---

│  │                          │      │                              │   │

## Prerequisites

### Docker path (recommended)
- Docker Engine 24+ and Docker Compose v2
- Linux host with the `wireguard` kernel module:
  ```bash
  sudo modprobe wireguard
  lsmod | grep wireguard          # must show output
  ```
- `>= 2 GB` RAM, `>= 4 GB` disk for logs and captures

### Native path
- Linux kernel 5.6+ (WireGuard built-in) or kernel module loaded
- `wireguard-tools`, `iproute2`, `tcpdump`, `gcc`, `jq`
- Python 3.9+ with `cryptography` and `scapy`:
  ```bash
  pip install cryptography scapy
  ```
- Root / `sudo` access

## Quick Start (Docker)

```bash
# 1 — Clone / copy the harness files into a working directory
mkdir wg-fuzz && cd wg-fuzz
# (place all harness files here)

# 2 — Ensure WireGuard module is loaded on the host
sudo modprobe wireguard

# 3 — Build images and start both containers
docker compose up --build

# 4 — Watch live anomalies in a second terminal
docker compose logs -f wg-fuzzer | grep '"anomaly": true'

# 5 — After the fuzzer finishes, export results for peer review
./export_results.sh
# → creates wg_fuzz_export_YYYYMMDD_HHMMSS.tar.gz
```

To run a single phase only:
```bash
PHASE=1 docker compose up --build
PHASE=2,3 docker compose up --build
```

To change mutation count or PRNG seed:
```bash
COUNT=100 SEED=1337 docker compose up --build
```

## Manual / Native Setup

```bash
# 1 — Start the server harness (creates isolated network namespace)
sudo ./wg_server_harness.sh start

# 2 — Run the fuzzer (reads config auto-generated by the harness)
sudo python3 wg_fuzzer.py --config wg_fuzz_config.json --phase all --count 50

# 3 — Watch kernel events in real time
sudo tail -f wg_fuzz_logs/monitor.jsonl | python3 -m json.tool

# 4 — Export results
./export_results.sh --native

# 5 — Tear down when done
sudo ./wg_server_harness.sh stop
```

## Fuzzing Phases

### Phase 1 — INIT (Handshake Initiation)

Targets `validate_header_len()`, `wg_cookie_validate_packet()`,
`wg_noise_handshake_consume_initiation()`, and `mix_dh()` in `noise.c`.

The fuzzer builds **real Noise IK packets** using X25519, ChaCha20Poly1305,
and HMAC-BLAKE2s — so MAC1 is valid and packets reach deeper code paths
than pure random mutation would achieve.

| Strategy | What it tests |
|---|---|
| `valid_baseline` | Sanity-check: server must respond |
| `wrong_type` | type-field dispatch in `wg_packet_receive()` |
| `truncation` | every boundary from 0 to 147 bytes |
| `extension` | `validate_header_len()` exact-length check |
| `mac1_corruption` | `wg_cookie_validate_packet()` rejection |
| `zeroed_fields` | NULL / zero ephemeral key handling |
| `all_ff_fields` | Field saturation |
| `torsion_ephemeral` | Low-order Curve25519 points → null-point check (ECC-2) |
| `bit_flip` | Random single/multi-bit mutations across packet |
| `sender_index_extreme` | `sender_index` boundary values (0, MAX, etc.) |
| `replay` | Anti-replay / `latest_timestamp` check |
| `flood` | Rate limiter — `INITIATIONS_PER_SECOND = 50` |
| `wrong_size_exact` | Sizes at ±1 and at other message type lengths |

### Phase 2 — SESSION (Data Packets)

Targets `decrypt_packet()`, `counter_validate()`, and
`wg_packet_consume_data_done()` in `receive.c`.

Because data packets are AEAD-protected, most mutations fail the
Poly1305 tag check. The session phase specifically probes:

| Strategy | What it tests |
|---|---|
| `wrong_key_idx` | `wg_index_hashtable_lookup()` — unknown key index |
| `counter_boundary` | Counter values at 0, REJECT\_AFTER\_MESSAGES ±1, U64\_MAX |
| `nonce_reuse` | Repeated counter — `counter_validate()` replay bitmap |
| `short_data` | Packets below `MESSAGE_MINIMUM_LENGTH = 32` |
| `oversized_data` | Payloads up to 128 KB |
| `bad_auth_tag` | Random payload — forces AEAD rejection path |
| `wrong_type_in_session` | Non-DATA types during session (type confusion) |
| `counter_window_edge` | Counter N then N − WINDOW\_SIZE − 1 (outside window) |

### Phase 3 — TEARDOWN (Session Disruption)

Targets the handshake state machine, cookie mechanism, and session
cleanup paths. Tests graceful and ungraceful session termination.

| Strategy | What it tests |
|---|---|
| `forced_rekey` | Valid initiations against an active session |
| `fake_cookie` | `wg_cookie_message_consume()` with garbage receiver_index |
| `fake_response` | `consume_response()` — unknown receiver_index lookup |
| `counter_wraparound` | Counters at REJECT\_AFTER\_MESSAGES, U64\_MAX, 0 |
| `rapid_reconnect` | Flood of valid initiations — rate limiter under load |
| `malformed_response` | TYPE=2 with correct length but all-corrupt fields |
| `state_confusion` | Out-of-order sequence: RESP → INIT → DATA → COOKIE → INIT |

## Anomaly Reference

The fuzzer flags events that deviate from expected server behaviour.
Any anomaly warrants manual investigation.

| Anomaly | Likely cause | Audit finding |
|---|---|---|
| Server sends RESPONSE to mutated init | MAC1 bypass, logic error | ECC-3 |
| Oversized response (> 296 bytes) | Memory disclosure | F1 (offset underflow) |
| Cookie reply to sub-16-byte packet | Length validation gap | F2 |
| Response faster than 0.5 ms to mutated packet | Short-circuit / crash-loop | — |
| `kernel_taint` changes during run | Kernel panic / KASAN / UBSAN | — |
| `kmsg` severity = CRASH | BUG/Oops/use-after-free detected | — |
| RESPONSE to torsion-point ephemeral | Null-point check missing | ECC-2 |

A `kernel_taint` of **0** means the kernel is clean. Any non-zero value
after a run means a kernel bug was triggered. Decode with:
```bash
python3 -c "
import sys
taint = int(sys.argv[1])
bits  = {0:'proprietary module',1:'forced module',2:'TAINT_UNSAFE_SMP',
         3:'SMP on wrong CPU',4:'machine check error',5:'bad page',
         6:'requested by user',7:'ACPI table overridden',8:'kernel died',
         9:'ACPI table overridden',10:'warn',11:'STAGING driver',
         12:'firmware workaround',13:'ex-t',14:'unsigned module',
         15:'soft lockup',16:'live patch',17:'test',18:'aux taint'}
for b, label in bits.items():
    if taint & (1 << b): print(f'  bit {b:2d}: {label}')
" YOUR_TAINT_VALUE
```

## Peer Review Export

```bash
./export_results.sh [--native] [--out DIR]
```

Produces a self-contained archive:

```
wg_fuzz_export_20240623_143022/
├── README_export.txt        # Run metadata (date, host, kernel, seed)
├── fuzzer.jsonl             # Complete fuzzer packet log
├── monitor.jsonl            # Kernel + packet monitor log
├── anomalies.jsonl          # Filtered: only anomaly=true events
├── capture.pcap             # Raw Wireshark-compatible capture
├── report.html              # Self-contained HTML report (no server needed)
└── manifest.sha256          # SHA-256 of every file above
```

The reviewer can:
- Open `report.html` directly in any browser — no web server needed
- Load `capture.pcap` in Wireshark and filter `udp.port == 51820`
- Verify file integrity with `sha256sum -c manifest.sha256`
- Re-run with the same seed: `SEED=42 docker compose up`

## Known Limitations

**Kernel module required on host.** Docker containers share the host kernel. The WireGuard
kernel module (`wireguard.ko`) must be loaded on the *host*, not in the container.
A userspace implementation (`wireguard-go`) will not exercise kernel code paths.

**Phase 2 without a live session.** The data-packet phase sends packets with
fake key indices. To fuzz the decryption path with a *valid* AEAD context, a full
Noise IK handshake must succeed first. The fuzzer establishes this automatically
when the server is reachable, but the `REJECT_AFTER_MESSAGES` and counter-window
strategies target the *pre-AEAD* checks regardless.

**MAC2 not tested.** `mac2` is only verified under load (when
`handshake_queue_len >= MAX_QUEUED_INCOMING_HANDSHAKES / 8`). The flood strategy
attempts to trigger this condition, but it depends on the server's core count and
load at runtime.

**Rate-limited strategies may not fire.** The `rapid_reconnect` and `flood`
strategies are throttled by `INITIATIONS_PER_SECOND = 50` on the server. Observe
`pkts_unknown` in the monitor stats to confirm packets are being dropped.
